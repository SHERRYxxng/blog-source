## 2. 通用规范

### 2.1 基础 URL

```
测试环境: https://oms-test.shsets.com
生产环境: https://oms.shsets.com
```

### 2.2 通用响应格式 (AjaxResult)

```json
{
  "code": 200,
  "msg": "操作成功",
  "data": { ... }
}
```

| code | 含义 |
|------|------|
| 200  | 成功 |
| 401  | 未授权/Token 失效 |
| 403  | 无权限 |
| 500  | 服务器内部错误 |

### 2.3 分页响应格式 (TableDataInfo)

```json
{
  "code": 200,
  "msg": "查询成功",
  "rows": [ ... ],
  "total": 100
}
```

### 2.4 认证方式

- **Header**: `Authorization: Bearer <token>`
- Token 通过 `/login` 接口获取
- JWT 格式，默认有效期 120 分钟

### 2.5 权限控制

使用 `@PreAuthorize("@ss.hasPermission('xxx:xxx:xxx')")` 注解控制接口权限。
当无权限时返回 403。

### 2.6 实体基类继承体系

```
BaseEntity
  ├── searchValue, remark, delFlag, createBy, createTime, updateBy, updateTime, params
  │
  └── SimpleEntity (id + createBy + createTime)
        │
        └── BaseModel (id + createBy + createTime + updateBy + updateTime)
              │
              ├── SmartModel (BaseModel + delFlag)
              │     ├── Event, ChangeProcess, Demand, Problem
              │     ├── SysUser, SysRole, SysDept
              │     ├── Contract, Device, Project
              │     └── ... (大部分业务实体)
              │
              └── BaseModel 直接子类:
                    ├── SysMenu, SysDictData, Notice
                    ├── MotorRoom, MotorDevices
                    └── ... (少数实体)
```

### 2.7 核心注解

| 注解 | 用途 |
|------|------|
| `@Log(title="xxx", businessType=INSERT/UPDATE/DELETE/EXPORT/IMPORT)` | 操作日志记录 |
| `@PreAuthorize("@ss.hasPermission('xxx:xxx:xxx')")` | 权限控制 |
| `@TableName(autoResultMap = true)` | MyBatis-Plus 表映射 |
| `@TableField(exist = false)` | 非数据库字段(查询参数/聚合字段) |
| `@TableField(typeHandler = ListToVarcharTypeHandler.class)` | 列表字段存储为逗号分隔字符串 |

## 3. 事件模块 (/operations/event)

### 3.1 save 发起事件 — 字段处理详解

**请求**: `POST /operations/event/save`  
**权限**: `operations:event:add`  
**请求体**: Event 实体 JSON

#### 处理流程 (EventServiceImpl.saveEvent):

```
saveEvent(Event event):
  ├── 1. ifSomeVariableChanged(event)  // 清理/修正字段
  ├── 2. event.setStatus(1)            // 强制设为 DRAFT(起草中) 状态
  │
  ├── [更新已有事件] event.id != null:
  │   ├── 3. 检查事件存在且状态为 DRAFT(1) 或 REJECTED(4)
  │   ├── 4. 检查当前用户为事件起草人(或拥有 representative 权限)
  │   ├── 5. 更新子事件关系
  │   └── 6. updateById(event)
  │
  └── [新建事件] event.id == null:
      ├── 3. 生成工单编号: I{yyyyMMdd}+{当日序号} 如 I20240520001
      ├── 4. save(event)  // 持久化
      ├── 5. 记录时间轴: timelineService.recordTimeLine(...)
      ├── 6. 保存子事件(subEventList)
      ├── 7. 保存关联关系(relationList):
      │       - 传入的 relation.type == "demand" 时自动创建双向关联
      └── 8. 创建待办: todoListService.save(...)
```

**工单编号生成规则**:
- 格式: `I{yyyyMMdd}{序号}`
- 示例: `I20240520001`
- 序号为当日事件数 + 1，使用雪花算法补充位宽

### 3.2 事件完整状态流转

| 状态码 | 枚举名 | 描述 | 操作 |
|--------|--------|------|------|
| 1 | DRAFT | 起草中 | 保存/提交 |
| 2 | APPROVAL | 待审核 | 审批通过/驳回 |
| 3 | PENDING_PROCESSING | 待处理 | 开始处理 |
| 4 | REJECTED | 已驳回 | 重新起草 |
| 5 | PROCESSING | 处理中 | 暂停/完成 |
| 6 | PAUSING | 暂停中 | 开始处理 |
| 7 | PENDING_TESTING | 待测试 | 开始测试 |
| 8 | TESTING | 测试中 | 测试完成 |
| 9 | PENDING_REVIEWING | 待复核 | 开始复核 |
| 10 | REVIEWING | 复核中 | 复核完成 |
| 11 | FAILED | 处理失败 | - |
| 12 | REOPENED | 已重开 | - |
| 13 | CLOSED | 已完成 | - |
| 14 | CANCELLED | 已取消 | - |
| 15 | ASSIGN | 转派 | 转派 |

**关键状态规则**:
- `canStart()`: DRAFT(1), REJECTED(4) — 允许提交
- `canBegin()`: PENDING_PROCESSING(3), PAUSING(6) — 允许开始处理
- 草稿+已驳回状态才可以更新事件内容
